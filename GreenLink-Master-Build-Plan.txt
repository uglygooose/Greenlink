APPROVED UX REBUILD PROGRAM
─────────────────────────────────────────────────────────────────────────────

Purpose:
- Preserve GreenLink visual identity
- Preserve backend truth and logic boundaries
- Remove dead weight, broken hierarchy, and low-value UX
- Rebuild toward a premium club operations OS centered on:
  - Tee Sheet
  - Finance / Close Day / Accounting handoff
  - Setup -> Configure -> Operate -> Close Day -> Report lifecycle

Execution principle:
- This is not a vanity new phase stack.
- This is a controlled rebuild program over the current baseline.
- Work should be tracked as rebuild slices, not as artificial long-range phase inflation.

Rebuild Slice R1 - Canonical doc alignment
- Merge repo truth with approved rebuild direction
- Update canonical files:
  - `CODEX-EXECUTION-RULES.txt`
  - `docs/MASTER_SYSTEM.md`
  - `SYSTEM_STATUS.md`
  - `GreenLink-Master-Build-Plan.txt`
- Success condition:
  - no blocking legacy rules remain
  - canonical docs preserve factual repo truth
  - rebuild direction is explicit and enforceable

Rebuild Slice R2 - Navigation and hierarchy reset
- Replace domain-first admin navigation with lifecycle-weighted navigation
- Target admin IA:
  - Today
  - Tee Sheet
  - Members
  - Finance
  - Performance
  - Operations (conditional)
  - Settings
- Remove or demote:
  - top-level POS emphasis
  - top-level Orders emphasis
  - weak "My Club" grouping logic
- Module/extension routes surface only when enabled and only where contextually justified
- Success condition:
  - top-level admin IA reflects product hierarchy
  - Tier 2/3 modules no longer distort Tier 1 flow

Rebuild Slice R3 - Today dashboard rebuild
- Replace summary-first dashboard weighting with operational work-queue weighting
- Today page must answer:
  - what needs action now
  - what blocks Close Day
  - what is off target
  - what requires tee-sheet or member intervention
- Remove low-value KPI duplication already covered elsewhere
- Success condition:
  - dashboard behaves like a shift-start control surface, not a report collage

Rebuild Slice R4 - Tee Sheet operational cockpit rebuild
- Preserve current backend-owned booking model and existing move/check-in actions
- Rebuild UX structure around:
  - compact operational header
  - virtualized canvas/grid
  - single context drawer
- Add native finance posture and actions into booking flow:
  - charges
  - payments
  - refunds
  - waivers / complimentary
- Reduce filter clutter; move complexity into compact presets/drawers
- Success condition:
  - tee sheet becomes the real daily operating surface

Rebuild Slice R5 - Settings / Golf setup rebuild
- Replace monolithic Golf Settings CRUD surface with guided configuration system
- Introduce:
  - setup order
  - validation state
  - draft vs active
  - publish / rollback
  - authority boundaries
- Preserve reusable editors where appropriate, but do not preserve weak page structure
- Success condition:
  - settings feel like controlled system setup, not technical admin clutter

Rebuild Slice R6 - Finance Close Day rebuild
- Finance default UX becomes Close Day-first
- Sequence:
  1. exceptions
  2. settlement / corrections
  3. batch generation
  4. reconciliation
  5. mapped export
  6. close audit
- Preserve existing batch/reconciliation backend foundations
- Improve client-accounting alignment UX
- Success condition:
  - finance acts as an operational close process, not a disconnected admin surface

Rebuild Slice R7 - Superadmin onboarding and accounting alignment
- Extend onboarding to support:
  - accounting export/template upload
  - profile mapping
  - readiness validation
  - structural ownership controls
- Superadmin owns:
  - module enablement
  - accounting structure alignment
  - go-live readiness
- Club admin owns:
  - bounded operational config
- Success condition:
  - onboarding drives real go-live readiness, not just form completion

Rebuild Slice R8 - Module demotion and extension model
- Reposition:
  - POS
  - Orders
  - Halfway
  - Pro Shop
  - Tennis
  - Bowls
as extensions, not system centerpieces
- Operations grouping should be conditional and module-driven
- Success condition:
  - extension modules add value without hijacking the OS

Rebuild Slice R9 - Performance and targets consolidation
- Rework reports + targets into a coherent performance surface
- Tie targets directly to:
  - Today alerts
  - Finance variances
  - operational interventions
- Remove isolated/weak target placement
- Success condition:
  - targets drive action rather than sit as a detached settings-like page

Rebuild Slice R10 - AI-assisted interventions
- Add AI-lite and rules-based assistance only where it improves actionability:
  - no-show risk
  - drift risk
  - target risk
  - gap-fill opportunities
- Suggestions must end in a next action, not informational fluff
- Success condition:
  - AI adds operational value instead of dashboard noise
