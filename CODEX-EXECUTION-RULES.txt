GREENLINK CODEX EXECUTION RULES
Last updated: 2026-04-03 18:45 SAST

Canonical role:
- Canonical implementation rules file.

Core rules
- Backend owns all logic.
- Frontend sends intent only.
- No duplicated logic across backend and frontend.
- No hidden side effects.
- No domain mixing.
- Narrow slices only. No broad rebuilds.
- Do not refactor unrelated code during correction work.

UI rules
- Benchmark UI files are the visual authority.
- Adapt benchmark structure; do not redesign it.
- Do not invent a new design system or new UI patterns during correction work.

Routing and shell rules
- Persistent shell is required for admin and superadmin workspaces.
- Admin and superadmin shells must be router-owned layout routes.
- `ProtectedRoute` wraps the layout, not individual pages.
- Admin and superadmin pages render content areas only.
- Do not reintroduce page-level `AdminShell` or `SuperadminShell`.
- All `/admin/*` routes, including `/admin/pos-terminal`, are nested inside `AdminLayout`. POS renders content area only.

Finance rules
- No frontend financial calculations.
- React must not compute finance totals, balances, arrears, revenue, outstanding values, transaction-volume KPIs, chart widths, or proportions.
- Finance KPI displays and chart bars must use backend read-model endpoints and their pre-computed pct fields only.
- Backend summary items carry `revenue_share_pct`, `volume_share_pct`, and `accounts_*_pct`; use these for bar widths, never derive them client-side.
- If a backend finance summary does not exist for a metric, remove the metric or show a neutral unavailable state.
- Do not preserve finance cards by recreating the same math elsewhere in React.

Superadmin onboarding rules
- Onboarding progression is backend-only.
- Frontend must not set current, next, or previous onboarding steps.
- Frontend must not set onboarding state directly.
- Frontend sends step intent only.
- Backend validates transitions, rejects invalid progression, and returns the resulting state.
- Do not introduce a generic workflow engine for onboarding.

Superadmin club management rules
- Pause and reactivate use PATCH `/superadmin/clubs/{id}/status` with `{ active: bool }`.
- Hard delete uses DELETE `/superadmin/clubs/{id}` and is blocked for live clubs (`registry_status == "active"`).
- Delete must clear non-cascading FK columns (`domain_event_record.club_id`, `platform_state.initial_club_id`) before deleting the club row.
- Delete confirmation modal is required in the UI before executing.
- Superadmin nav contains only live routes; do not add placeholder items that link to the same page.
- Superadmin overview and clubs actions must carry a concrete club selection into the destination route; do not drop users into unscoped admin pages when the action is club-specific.
- Superadmin may bridge into existing club-scoped `/admin/*` workspaces after selecting a club; do not duplicate finance, rules, or dashboard surfaces inside superadmin when the admin workspace already exists.

Tee sheet and booking rules
- Tee sheet is a read model.
- Do not write tee-sheet state directly from the frontend.
- Booking lifecycle transitions remain backend-owned.

Orders and payment rules
- Orders are not payments.
- Order placement, charge posting, and settlement recording remain distinct concerns.

Player truth rules
- No fake booking data in player UI.
- Do not create placeholder booking arrays to fill empty states.
- If a backend member-booking read model does not exist, show loading or empty state only.

Documentation rules
- `docs/MASTER_SYSTEM.md` is the canonical current system definition.
- `GreenLink-Master-Build-Plan.txt` is the canonical phase/slice plan.
- `CODEX-EXECUTION-RULES.txt` is the canonical implementation-rules file.
- `SYSTEM_STATUS.md` is the canonical current-state snapshot.
- Remove, align, or clearly archive any overlapping doc that conflicts with these files.
